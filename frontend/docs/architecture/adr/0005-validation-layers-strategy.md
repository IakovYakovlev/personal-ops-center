# ADR 0005: Стратегия многоуровневой валидации

- **Статус:** Proposed
- **Дата:** 2026-02-26
- **Автор:** Iakov
- **Область (Scope):** Global

## Контекст (Context)

В системе есть несколько точек входа, где пользовательские данные могут быть некорректными (размер файла, тип, контент, формат). Если валидация выполняется только на одном уровне, возрастает риск ухудшения UX, роста нагрузки и появления уязвимостей. Нужна единая стратегия, которая описывает, где и зачем выполняется проверка данных.

## Варианты решения (Considered Options)

1. **Вариант А:** Валидация только на фронтенде (быстро, но небезопасно).
2. **Вариант Б:** Валидация только в сервисе API (лучше, но нет UX-подсказок).
3. **Вариант В:** Многоуровневая валидация (Frontend + Service + Backend).

## Решение (Decision Outcome)

**Выбран Вариант В.** Мы используем многоуровневую валидацию:

1. **Frontend (UX layer):** быстрые проверки и подсказки пользователю до отправки запроса.
2. **Client Service (Safety layer):** повторная проверка перед сетевым запросом (defense in depth).
3. **Backend (Security layer):** финальная обязательная валидация, которой можно доверять.

### Обоснование (Pros)

- Улучшение UX: быстрые ошибки без лишних запросов.
- Снижение нагрузки на API за счет раннего отсечения невалидных запросов.
- Повышение безопасности: сервер не доверяет клиенту.
- Defense-in-depth снижает риск обхода ограничений.

### Недостатки (Cons)

- Дублирование логики валидации.
- Дополнительные точки поддержки при изменении правил.

## Последствия (Consequences)

- Валидация должна существовать на трех уровнях (UI, client-service, backend).
- Правила валидации фиксируются в документации, чтобы избежать рассинхронизации.
- При изменении лимитов или форматов требуется обновление всех уровней.
